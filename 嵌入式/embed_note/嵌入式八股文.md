## 1 MCU 上电后 main 函数之前发生了什么？
```
上电复位
↓
从复位向量取启动地址
↓
执行启动文件 startup
↓
初始化栈指针
↓
拷贝 .data 段到 RAM
↓
清零 .bss 段
↓
SystemInit，配置时钟
↓
调用 main
```

| 段         | 含义                            |
| --------- | ----------------------------- |
| `.text`   | 代码段，通常在 Flash                 |
| `.rodata` | 只读常量                          |
| `.data`   | 已初始化全局/静态变量，运行在 RAM，初值在 Flash |
| `.bss`    | 未初始化全局/静态变量，启动时清零             |
| heap      | 动态分配区                         |
| stack     | 函数调用栈                         |
## 2 常用内存单元
| 类型           | 掉电是否保留 | 主要用途           | 速度   | 写入寿命  | 常见接口/位置            |
| ------------ | ------ | -------------- | ---- | ----- | ------------------ |
| SRAM         | 不保留    | 运行时变量、栈、堆      | 很快   | 基本无限  | MCU 内部             |
| DRAM / SDRAM | 不保留    | 大容量运行内存        | 快    | 基本无限  | 外部并行总线             |
| PSRAM        | 不保留    | 图像缓存、UI 缓冲、大数组 | 中等   | 基本无限  | SPI/QSPI/Octal SPI |
| ROM          | 保留     | 固化程序、BootROM   | 只读   | 不可改   | MCU 内部             |
| OTP          | 保留     | 一次性配置、校准、密钥    | 读快   | 只能写一次 | MCU 内部             |
| Flash        | 保留     | 程序、固件、大容量数据    | 读快写慢 | 有擦写寿命 | 内部/外部 SPI/QSPI     |
| EEPROM       | 保留     | 参数、配置、小数据      | 慢    | 有写入寿命 | I2C/SPI/内部         |
| FRAM         | 保留     | 高频参数、计数、日志     | 快    | 寿命高   | I2C/SPI            |
| MRAM         | 保留     | 工业高可靠存储        | 快    | 寿命高   | SPI/并行             |
| eFuse        | 保留     | 芯片配置、加密密钥      | 读快   | 一次性熔断 | MCU 内部             |
## 3 flash操作
### 3.1 内部flash
MCU 内部 Flash 通常放：

```
程序代码
常量
默认参数
少量配置
Bootloader
App 固件
```

程序里的：

```c
const char name[] = "device";
```

通常在 Flash 的 `.rodata` 段。

代码本身在：

```
.text 段
```

### 3.2 外部flash
外部 Flash 常见接口：
```
SPI Flash
QSPI Flash
OSPI Flash
Parallel NOR Flash
NAND Flash
```

常用于：

```
大容量固件
OTA 分区
文件系统
字库
图片
资源
日志
```

ESP32 的程序通常就在外部 SPI Flash 里。

### 3.3 flash规则
Flash 擦除后通常是：
```
0xFF
```
写入只能：
```
1 → 0
```
不能直接：
```
0 → 1
```
所以要从 `0` 变回 `1`，必须擦除。

## 4 `#pragma once`的作用是什么有必要吗
  
`#pragma` once 是头文件保护的非标准但广泛支持的指令，功能等同于传统的 include guard：
```c
// #pragma once 等价于
#ifndef DEV_AUDIO_H
#define DEV_AUDIO_H
// ... 头文件内容 ...
#endif
```

## 5 freertos临界区如何实现
FreeRTOS 提供了两组宏来进入/退出临界区：

| 适用范围    | 进入临界区                           | 退出临界区                             | 原理                                                 |
| ------- | ------------------------------- | --------------------------------- | -------------------------------------------------- |
| **任务级** | `taskENTER_CRITICAL()`          | `taskEXIT_CRITICAL()`             | 通过 `ulPortSetInterruptMask()` 关中断，并维护一个嵌套计数（可嵌套调用） |
| **中断级** | `taskENTER_CRITICAL_FROM_ISR()` | `taskEXIT_CRITICAL_FROM_ISR( x )` | 只关闭优先级较低的或全部可配置的中断（具体由移植层决定）                       |

**核心特点**：
- 基于**关中断**实现，进入临界区后不会被其他任务或中断打断（除非是最高优先级、不可屏蔽的中断）。
    
- 执行速度快，适合保护**极短的、不能有任何上下文切换**的操作，例如修改全局变量、链表节点操作。
    
- **不能**睡眠、阻塞或等待任何事件（因为关中断后无法响应系统 tick，调度器也会被冻结）。
实现细节:
- 1 
```c
step1:
portDISABLE_INTERRUPTS()
//关中断等价__set_BASEPRI(configMAX_SYSCALL_INTERRUPT_PRIORITY);
//例如设置BASEPRI寄存器为0x50
//优先级 >= 0x50 的中断被屏蔽
//优先级 < 0x50 的高优先级中断继续运行
step2:
uxCriticalNesting++;
//增加嵌套计数,关中断允许嵌套
//只有uxCriticalNesting=0才会重启中断
step3:
if( uxCriticalNesting == 1 )  
{  
	configASSERT(  
	( portNVIC_INT_CTRL_REG &  
	portVECTACTIVE_MASK ) == 0  
	//判断当前是否运行在中断
);  
}
```

## 6 freertos中的互斥锁和信号量
### 一、**本质区别**

信号量（Semaphore）
信号量本质是：

> **同步工具**

用于通知某个事件发生了。

---

互斥锁（Mutex）
互斥锁本质是：

> **资源保护工具**

用于保证同一时间只有一个任务访问共享资源。
### 二、**拥有者（Owner）概念**

Semaphore
- 没有拥有者
- 可以在不同任务take和give
Mutex
- xSemaphoreTake(mutex);会把当前任务注册为拥有者
- xSemaphoreGive(mutex);只有拥有者才能释放
### 三、**优先级继承（Priority Inheritance）**
优先级反转问题
如果有:
```
高优先级任务 H
中优先级任务 M
低优先级任务 L
```

时刻1
L获得Mutex：
```
xSemaphoreTake(mutex);
```

时刻2
H想获取Mutex：
```
xSemaphoreTake(mutex);
```

发现被占用：

```
H Block
```
时刻3
M开始运行

由于：
```
H被阻塞
L优先级低
```

结果：
```
M一直运行
```

L得不到CPU。
结果
```
H等L释放锁
L等CPU
CPU被M占着
```
形成：

```
Priority Inversion优先级反转
```
Mutex解决方案:
```
Priority Inheritance
优先级继承:
当L持有H高优先级任务等待的mutex,L任务的优先级会临时提升到H
```

## 7 Task Notify为什么轻量化
一句话：

> **Task Notify 轻量，是因为它直接使用任务 TCB 里的成员变量实现通知，不需要额外创建内核对象。**

对比一下：

|机制|是否需要额外对象|底层|
|---|---|---|
|Semaphore|需要|Queue对象|
|Mutex|需要|Queue对象|
|EventGroup|需要|EventGroup对象|
|Task Notify|不需要|TCB成员变量|

例如：

```c
TaskA
 └─ TCB
     ├─ priority
     ├─ stack
     ├─ notify_value
     └─ notify_state
```

通知时：

```c
vTaskNotifyGive(TaskA);
```

本质就是：

```c
TaskA->notify_value++;
TaskA->notify_state = notified;
```

然后唤醒任务。

而 Semaphore 需要：

```text
Semaphore对象
    ↓
Queue控制块
    ↓
等待链表
    ↓
资源管理
    ↓
唤醒任务
```

中间多了一层内核对象。

所以 Task Notify 具有：

- 更少 RAM 占用（通常不需要额外内存）
    
- 更少 CPU 开销
    
- 更快的 ISR → Task 通知路径
在 FreeRTOS 官方文档中，Task Notification 被称为：

> **A lightweight and fast direct-to-task communication mechanism**（轻量且快速的直接任务通信机制）。
