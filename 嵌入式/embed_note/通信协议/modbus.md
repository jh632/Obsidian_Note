## 1 modbus硬件基础
### 1 硬件接线

RS-485 使用**差分传输**，通常有两根线：

|差分电压|逻辑状态|
|---|---|
|VA - VB > +200 mV|一种逻辑状态|
|VA - VB < -200 mV|另一种逻辑状态|
RS485抗干扰强,差分信号

### 2 接线方式

```
A ---------------- A ---------------- A
B ---------------- B ---------------- B
GND -------------- GND -------------- GND
```

### 3 RS-485 和 UART 的关系
|层级|作用|
|---|---|
|UART|定义起始位、数据位、校验位、停止位|
|RS-485|定义电气传输方式|
|Modbus RTU|定义地址、功能码、寄存器、CRC|

### 4 RS-485 半双工方向控制
两线制rs485需要控制方向

```
RO  -> MCU RX
DI  <- MCU TX
DE  <- 发送使能
/RE <- 接收使能
A/B -> 总线
```

```
DE 和 /RE 接在一起，由一个 GPIO 控制
```

| GPIO                                                | 状态   |
| --------------------------------------------------- | ---- |
| 0                                                   | 接收模式 |
| 1                                                   | 发送模式 |
### 5 多设备总线结构
RS485支持一主多从
```
      主机
       |
A -----+----- 从机1 ----- 从机2 ----- 从机3
B -----+----- 从机1 ----- 从机2 ----- 从机3
```

例如 Modbus RTU 协议：

```
主机：问 01 号设备的 40001 寄存器从机：01 号设备回答其他设备：不响应
```

RS-485 本身没有“谁能说话”的仲裁机制，所以必须由上层协议控制，避免多个设备同时发送。

如果两个设备同时发，会产生：

```
总线冲突
```

表现为：

- CRC 错误；
- 数据乱码；
- 丢帧；
- 通信偶发失败；
- 某些收发器发热或保护。
### 6 终端电阻
RS-485 是长线高速传输，线缆可以看成传输线。为了减少信号反射，通常在总线两端加终端电阻。

典型值：

```
120 Ω
```

接法：

```
A ----[120Ω]---- B
```

注意是：

> 只在总线最两端加，不是每个设备都加。


正确示例：

```
[120Ω] 主机 ---- 从机1 ---- 从机2 ---- 从机3 [120Ω]
```

### 7 GND需求
理论上 RS-485 是差分信号，不是靠 GND 传输数据。
如果两个设备地电位差太大，虽然 A/B 是差分，但接收器输入可能超出范围，导致：

- 通信异常；
- 芯片损坏；
- ESD 保护二极管导通；
- 总线不稳定。

所以工程上常用：

```
A/B + 信号地
```

远距离、强干扰环境还会加：

- TVS 管；
- 共模电感；
- 隔离 RS-485 芯片；
- 屏蔽双绞线；
- 单点接地或合理接地策略。
## 2 modbus RTU通信协议
### 1 帧格式
```
[从机地址][功能码][数据区][CRC低字节][CRC高字节]
```

|字段|长度|作用|
|---|--:|---|
|从机地址|1 字节|指定访问哪个设备|
|功能码|1 字节|指定读还是写、读哪类寄存器|
|数据区|N 字节|寄存器地址、数量、写入值等|
|CRC|2 字节|错误校验，低字节在前|

```
01 03 00 00 00 02 C4 0B
```

含义是：

|字节|含义|
|---|---|
|`01`|从机地址 1|
|`03`|功能码 03，读保持寄存器|
|`00 00`|起始寄存器地址 0|
|`00 02`|读取 2 个寄存器|
|`C4 0B`|CRC16|

这句话的意思：

```
问 1 号设备：从地址 0 开始，读 2 个保持寄存器。
```

### 2 crc16

特点：

- 多项式：`0xA001`
- 初值：`0xFFFF`
- CRC 结果低字节在前，高字节在后
```c
#include <stdint.h>
#include <stddef.h>

uint16_t modbus_crc16(const uint8_t *data, size_t len)
{
    uint16_t crc = 0xFFFF;

    for (size_t i = 0; i < len; i++) {
        crc ^= data[i];

        for (int bit = 0; bit < 8; bit++) {
            if (crc & 0x0001) {
                crc = (crc >> 1) ^ 0xA001;
            } else {
                crc >>= 1;
            }
        }
    }

    return crc;
}
```

### 3. 异常响应

如果从机不能正常处理请求，会返回异常帧。

格式：

```
[从机地址][功能码 | 0x80][异常码][CRC]
```

例如请求功能码是：

```
03
```

异常响应功能码就是：

```
83
```

例子：

```
01 83 02 C0 F1
```

含义：

|字段|含义|
|---|---|
|`01`|从机地址|
|`83`|03 功能码的异常响应|
|`02`|异常码|
|`C0 F1`|CRC|

常见异常码：

|异常码|含义|
|---|---|
|01|非法功能码|
|02|非法数据地址|
|03|非法数据值|
|04|从机设备故障|
|06|从机忙|

如果你读寄存器经常返回 `83 02`，通常说明：

```
寄存器地址错了，或者功能码选错了。
```

### 4 RTU 的帧间隔

Modbus RTU 没有像 TCP 那样的长度头完整描述整帧，也没有固定结束符。

它依靠**空闲时间间隔**区分帧。

经典规则：

```
一帧内部字符间隔不能超过 1.5 个字符时间两帧之间至少间隔 3.5 个字符时间
```

字符时间和波特率有关。

例如 UART 配置：

```
9600 baud, 8N1
```

一字符大概：

```
1 起始位 + 8 数据位 + 1 停止位 = 10 bit10 / 9600 ≈ 1.04 ms
```

所以：

```
3.5 字符时间 ≈ 3.65 ms
```

工程上常见做法：

- 主机发完请求后等待响应；
- 从机收到完整帧后延迟很短时间再回复；
- 接收端用 UART idle 中断、定时器或超时来判断一帧结束。

在 mcu 上，常见实现是：

```
UART 接收中断 / DMA 接收+空闲超时判断一帧结束+CRC 校验+解析功能码
```
