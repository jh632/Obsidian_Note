# 观察者模式 / 回调模式

## 1. 一句话

> Subject（主题）状态变化时，主动通知所有已注册的 Observer（观察者）——不用轮询，不用耦合。

## 你已经会了，只是还没命名

在你看这篇笔记之前，你已经在多个项目里写了观察者模式——只是叫它"回调函数"或"中断通知"：

| 你写的代码 | 隐含的观察者模式 |
|---|---|
| `esp_ble_gap_register_callback(gap_cb)` | BLE GAP 事件 → 单一回调（[[blueteeth]]） |
| `esp_ble_gatts_register_callback(gatts_cb)` | BLE GATTS 事件 → 单一回调（[[blueteeth]]） |
| `__interrupt void xint1_isr(void)` → 通知任务 | GPIO 中断 → 任务通知（[[TI API]]） |
| `__interrupt void canAISR(void)` → `xQueueSendFromISR` | CAN 中断 → 队列 → CAN RX 任务（[[can总线]]） |
| `xSemaphoreGiveFromISR` → 任务唤醒 | 二值信号量作为 1-bit 通知（[[TI RTOS和FreeRTOS]]） |

**观察者模式不是新东西——是把你在项目里已经在做的"回调注册 → 事件通知"命名化、规范化。**

## 2. 核心概念

```
                    register(observer)        notify(event)
    ┌──────────┐  ◄───────────────────  ┌──────────────┐
    │ Observer │                        │   Observer   │
    │    1     │  ──────────────────►   │      2       │
    └──────────┘       update()         └──────────────┘
         ▲                                    ▲
         │           notify(event)            │
         │    ┌─────────────────────┐         │
         └────│      Subject        │─────────┘
              │  - state            │
              │  - observer_list[]  │──────► Observer N
              │  + register()       │
              │  + unregister()     │
              │  + notify_all()     │
              └─────────────────────┘
```

四个要素：

| 概念      | 英文       | 在 C 中的表示                                       | 示例                                          |
| ------- | -------- | ---------------------------------------------- | ------------------------------------------- |
| **主题**  | Subject  | 持有观察者列表的结构体                                    | `battery_monitor_t`                         |
| **观察者** | Observer | 函数指针 `typedef void (*observer_cb)(event, ctx)` | `display_update()`, `ble_notify()`          |
| **注册**  | Register | 把函数指针加入数组/链表                                   | `battery_register_observer(display_update)` |
| **通知**  | Notify   | 遍历数组/链表，逐个调用                                   | `for(i=0;i<n;i++) observers[i](evt)`        |

**观察者模式的本质**：把"谁来响应事件"从硬编码变成运行时注册——Subject 不知道 Observer 是谁，只知道"有人关心我的变化"。

### 推 vs 拉

| 方式          | Subject 做什么 | Observer 做什么    | 适用场景                    |
| ----------- | ----------- | --------------- | ----------------------- |
| **推（Push）** | 通知时带上所有数据   | 被动接收            | 数据量小、所有 Observer 需要相同数据 |
| **拉（Pull）** | 只通知"有变化"    | 主动回读 Subject 状态 | 数据量大、各 Observer 需要不同数据  |

嵌入式里**推模式占 90%**：事件 + 少量数据直接传过去，避免 Observer 回读的开销。

## 3. Observer vs Pub-Sub

你在 [[嵌入式设计模式总览]] 里已经看到了这个区分，这里展开讲清楚：

```
Observer（观察者）                  Pub-Sub（发布-订阅）

┌──────────┐                       ┌──────────┐
│ Subject  │──通知──► Observer1    │Publisher │──发布──► EventBus
│          │──通知──► Observer2    │          │           │
│          │──通知──► Observer3    └──────────┘     ┌──────┴──────┐
└──────────┘                                       │   EventBus  │
     ▲                                             │ - topic 路由 │
     │                                             │ - 消息缓存   │
  Subject 直接持有                               └──────┬──────┘
  Observer 列表                                          │
                                                    ┌────┴────┐
                                                    ▼         ▼
                                               Subscriber1 Subscriber2
```

| 维度       | Observer                | Pub-Sub                            |
| -------- | ----------------------- | ---------------------------------- |
| **耦合度**  | Subject 知道 Observer 接口  | 双方只知道 EventBus                     |
| **通知路径** | Subject → Observer 直接调用 | Publisher → EventBus → Subscriber  |
| **注册位置** | Observer 向 Subject 注册   | Subscriber 向 EventBus 订阅 topic     |
| **C 实现** | 函数指针数组/链表               | 消息队列 + topic 路由表                   |
| **典型场景** | GPIO 中断回调数组、传感器数据通知     | CAN 总线消息分发、MQTT topic 订阅、BLE 特征值广播 |
| **开销**   | 低（直接函数调用）               | 中（经过中间层）                           |
| **跨任务**  | 同一任务上下文                 | 天然跨任务/跨核                           |

**一句话选择**：
- 同一模块内、同步通知 → **Observer**（函数指针数组）
- 跨模块、跨任务、一对多广播 → **Pub-Sub**（消息队列/EventBus）

嵌入式里两者经常混用：底层用 Observer 做驱动回调，上层用 Pub-Sub 做跨模块通信。

## 4. C 语言三种实现

从你最熟悉的单一回调出发，逐步演进到多观察者注册。

### 4.1 单一回调函数指针（退化版 Observer）

这就是你在 BLE 代码里已经在用的模式——ESP-IDF 的 `esp_ble_gap_register_callback(gap_cb)` 本质上就是注册一个全局回调：

```c
// Subject: 电池电量传感器
typedef enum { BATT_NORMAL, BATT_LOW, BATT_CRITICAL } batt_level_t;

typedef void (*batt_callback_t)(batt_level_t level, int voltage_mv);

static batt_callback_t g_callback = NULL;  // 只有一个观察者

void batt_register_callback(batt_callback_t cb) {
    g_callback = cb;
}

// ADC 采样完成后调用（可能在 ISR 中）
void batt_on_sample_done(int voltage_mv) {
    batt_level_t level;
    if (voltage_mv > 3600)      level = BATT_NORMAL;
    else if (voltage_mv > 3300) level = BATT_LOW;
    else                        level = BATT_CRITICAL;

    if (g_callback) {
        g_callback(level, voltage_mv);  // 通知唯一的观察者
    }
}
```

**优点**：极简，零开销（一个函数指针）。  
**缺点**：只能有一个观察者。注册新回调会覆盖旧的——如果你同时需要显示屏更新和 BLE 通知，这就炸了。

**适用场景**：硬件中断回调、驱动层单一消费者（如 BLE GAP 事件——协议栈只有一个 GAP 处理者）。

**这就是你当前的舒适区——你已经在写了，只是不知道它叫"退化版观察者"。**

### 4.2 回调数组 —— 多观察者静态注册（主力方法）

从"一个回调"到"一组回调"，两步之遥：

```c
#define MAX_OBSERVERS 4

typedef void (*batt_observer_t)(batt_level_t level, int voltage_mv);

static batt_observer_t observers[MAX_OBSERVERS];
static int observer_count = 0;

bool batt_register_observer(batt_observer_t obs) {
    if (observer_count >= MAX_OBSERVERS) return false;
    observers[observer_count++] = obs;
    return true;
}

void batt_unregister_observer(batt_observer_t obs) {
    for (int i = 0; i < observer_count; i++) {
        if (observers[i] == obs) {
            // 紧凑化：把最后一个移到当前位置
            observers[i] = observers[--observer_count];
            return;
        }
    }
}

void batt_notify_all(batt_level_t level, int voltage_mv) {
    for (int i = 0; i < observer_count; i++) {
        observers[i](level, voltage_mv);
    }
}

// 使用时注册多个观察者
batt_register_observer(display_update);   // 更新屏幕电池图标
batt_register_observer(ble_battery_notify); // BLE 通知手机
batt_register_observer(low_power_alert);  // 低电量报警
```

**注销技巧**：不保持顺序时用"末尾填充空洞"（O(1)）；需要保持顺序时用 `memmove`（O(n)，但嵌入式里 n 通常 ≤ 4，不差这点）。

**优点**：简单、确定性强、无动态分配。  
**缺点**：观察者数量有上限、注销时遍历开销（n 很小可忽略）。

**适用场景**：90% 的嵌入式观察者需求。传感器数据分发、电池状态通知、按键事件分发。

### 4.3 侵入式链表 —— 动态注册/注销

当观察者数量不固定、需要频繁增删时，静态数组不够用。用 [[侵入式链表]] 把观察者节点嵌入到各自的结构体里：

```c
// 观察者链表节点（侵入式——嵌入到观察者自己的结构体里）
typedef struct batt_observer_node {
    struct batt_observer_node *next;
    void (*update)(batt_level_t level, int voltage_mv);
} batt_observer_node_t;

static batt_observer_node_t *observer_list = NULL;  // 链表头

void batt_register_observer(batt_observer_node_t *node) {
    // 头插法——O(1)，不关心顺序
    node->next = observer_list;
    observer_list = node;
}

void batt_unregister_observer(batt_observer_node_t *target) {
    batt_observer_node_t **indirect = &observer_list;
    while (*indirect) {
        if (*indirect == target) {
            *indirect = target->next;  // 从链表中摘除
            return;
        }
        indirect = &(*indirect)->next;
    }
}

void batt_notify_all(batt_level_t level, int voltage_mv) {
    for (batt_observer_node_t *p = observer_list; p; p = p->next) {
        p->update(level, voltage_mv);
    }
}
```

**侵入式链表的优势**：节点内存在观察者自己的结构体里，零额外分配。「注册」只是插指针，「注销」只是摘指针。你在 [[侵入式链表]] 笔记里已经理解了原理，Linux 内核和 FreeRTOS 到处都是这种模式。

**适用场景**：观察者数量不确定、动态增减频繁、每个观察者有独立生命周期。

### 4.4 RTOS 任务通知 —— 任务作为观察者

前面三种都是"函数回调"——同步调用，在 Subject 的上下文中执行。当观察者是独立的任务时，需要**异步通知**：

```c
/* ===== Subject 端（可能在 ISR 中） ===== */
void batt_on_sample_done_from_isr(int voltage_mv) {
    batt_level_t level = batt_calc_level(voltage_mv);

    // 通知显示屏任务
    xTaskNotifyFromISR(display_task_handle, level, eSetValueWithOverwrite, NULL);
    // 通知 BLE 任务
    xTaskNotifyFromISR(ble_task_handle, level, eSetValueWithOverwrite, NULL);
    // 通知低功耗监控任务
    xTaskNotifyFromISR(power_task_handle, level, eSetValueWithOverwrite, NULL);
}

/* ===== Observer 端（各自的任务循环中） ===== */
void display_task(void *pv) {
    uint32_t level;
    while (1) {
        // 等待通知（阻塞，不浪费 CPU）
        if (xTaskNotifyWait(0, ULONG_MAX, &level, portMAX_DELAY) == pdTRUE) {
            display_update_battery((batt_level_t)level);
        }
    }
}
```

对比三种通知机制：

| 机制 | 适用场景 | 携带数据 | ISR 安全 |
|---|---|---|---|
| **直接任务通知** `xTaskNotify` | 一对一通知（知道具体任务句柄） | 32-bit 值 | `xTaskNotifyFromISR` ✓ |
| **队列** `xQueueSend` | 一对多、需要缓冲多条消息 | 任意结构体 | `xQueueSendFromISR` ✓ |
| **事件组** `xEventGroupSetBits` | 一对多、多条件组合等待 | 24-bit 标志位 | `xEventGroupSetBitsFromISR` ✓ |

**选择原则**：一对一用任务通知（最快），一对多用队列（最灵活），多条件等待用事件组。

**RTOS 通知 vs 函数回调**：

| 维度 | 函数回调（4.2/4.3） | RTOS 通知（4.4） |
|---|---|---|
| **执行上下文** | Subject 的上下文（任务或 ISR） | Observer 自己的任务上下文 |
| **时序** | 同步（立即执行） | 异步（等 Observer 任务调度） |
| **阻塞** | 回调不能阻塞 | Observer 可以阻塞等待 |
| **适用** | 轻量、快速、同任务内 | 重处理、跨任务、需要解耦时序 |

**你在 CAN 总线的架构里已经在用 RTOS 通知模式了**：CAN ISR → `xQueueSendFromISR` → CAN RX 任务。ISR 是 Subject，CAN RX 任务是 Observer，队列是通知通道。

## 5. 嵌入式特有考量

### 5.1 ISR 中的观察者通知

中断里通知观察者，最大的约束是：**不能阻塞、不能等锁、不能调非 FromISR 的 API**。

```c
/* ===== 裸机：直接在 ISR 中调回调（简单但危险） ===== */
// 只适合回调极短、无锁的场景
__interrupt void adc_isr(void) {
    int voltage = ADC_readResult(ADC_BASE);
    ADC_clearInterrupt(ADC_BASE);

    // 直接在 ISR 上下文调所有观察者
    batt_notify_all(batt_calc_level(voltage), voltage);  // 回调必须极快！
}

/* ===== RTOS：延迟到任务上下文 ===== */
// 推荐做法——ISR 只做最小工作，通知放到任务里
static QueueHandle_t batt_queue;  // 在初始化时 xQueueCreate(8, sizeof(int))

__interrupt void adc_isr(void) {
    int voltage = ADC_readResult(ADC_BASE);
    ADC_clearInterrupt(ADC_BASE);

    BaseType_t xHigherPriorityTaskWoken = pdFALSE;
    xQueueSendFromISR(batt_queue, &voltage, &xHigherPriorityTaskWoken);
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);  // 如果需要切换
}

// 任务中安全地通知所有观察者
void batt_task(void *pv) {
    int voltage;
    while (1) {
        if (xQueueReceive(batt_queue, &voltage, portMAX_DELAY)) {
            batt_notify_all(batt_calc_level(voltage), voltage);  // 安全调用
        }
    }
}
```

**ISR 中直接调回调的唯一安全条件**：
1. 回调中无循环/无阻塞（< 几微秒）
2. 回调中不访问被中断的任务可能也在访问的数据（无锁竞争）
3. 回调不调 RTOS API（除非 FromISR 版本）

### 5.2 观察者执行顺序

数组版保持注册顺序，链表版取决于插入顺序（头插法 = 逆序）。多数场景不需要严格顺序，但如果 BLE 通知必须在显示屏更新之后：

```c
// 方案 1：按优先级排序注册（约定优于配置）
// 先注册低优先级，后注册高优先级（如果有遍历序依赖）

// 方案 2：数组 + 优先级字段
typedef struct {
    void (*update)(batt_level_t, int);
    int priority;  // 数值越小越先执行
} prio_observer_t;

static prio_observer_t observers[MAX_OBS];
static int observer_count = 0;

bool batt_register_observer_prio(void (*update)(batt_level_t, int), int prio) {
    if (observer_count >= MAX_OBS) return false;
    // 按优先级插入（保持有序）
    int pos = observer_count;
    while (pos > 0 && observers[pos-1].priority > prio) {
        observers[pos] = observers[pos-1];
        pos--;
    }
    observers[pos] = (prio_observer_t){update, prio};
    observer_count++;
    return true;
}
```

不过说实话——**嵌入式里 90% 的场景不需要优先级排序**。4 个观察者，顺序无所谓。

### 5.3 内存策略

| 策略 | 适用场景 | 代价 |
|---|---|---|
| **静态数组** `static obs_t arr[MAX]` | 观察者数量编译时已知 | 略占 BSS（~几十字节） |
| **侵入式链表** | 观察者数量不固定 | 节点内存由观察者自己承担 |
| **动态分配** `malloc` | 几乎不用 | 碎片、失败处理 |

**嵌入式推荐**：静态数组 > 侵入式链表 >>> 动态分配。你的 [[嵌入式存储器体系速查笔记]] 已经解释了为什么——`.bss` 段的空间是编译时就确定的，零运行时开销。

## 6. 常见陷阱

### 6.1 在通知过程中注销观察者

```c
/* ===== 经典 bug：遍历中删除 ===== */
void batt_notify_all(batt_level_t level, int voltage) {
    for (int i = 0; i < observer_count; i++) {
        observers[i](level, voltage);
        // 如果 observers[i] 内部调了 batt_unregister_observer(observers[i])
        //    注销函数里把末尾移过来，observer_count--, i 不变
        //    新的 observers[i] 可能被跳过，或者末尾搬过来的被调了两次
    }
}

/* ===== 修复：标记删除 ===== */
static bool observer_active[MAX_OBS];
static observer_t observers[MAX_OBS];
static int observer_count;

void batt_notify_all(batt_level_t level, int voltage) {
    for (int i = 0; i < observer_count; i++) {
        if (observer_active[i]) {
            observers[i](level, voltage);
        }
    }
    // 通知完了再清理
    int write = 0;
    for (int i = 0; i < observer_count; i++) {
        if (observer_active[i]) {
            observers[write] = observers[i];
            observer_active[write] = true;
            write++;
        }
    }
    observer_count = write;
}
```

### 6.2 回调中修改 Subject 状态导致递归通知

```c
// Observer 收到低电量通知后，调了 batt_set_sample_period(100ms)
// batt_set_sample_period 内部又调了 batt_notify_all(...)
// -> 递归通知 -> 栈溢出
```

**修复**：加 `notify_in_progress` 标志，或推迟状态变更到通知完成之后。

### 6.3 ISR 中调用了非 FromISR 的 RTOS API

你已经在 [[TI RTOS和FreeRTOS]] 笔记中写过——FreeRTOS 里 ISR 只能调 `FromISR` 版本的 API，这是硬规则。

### 6.4 回调函数跑飞

观察者是个函数指针——如果传了空指针或野指针：

```c
// 防御式调用
if (observers[i] != NULL) {
    observers[i](level, voltage);
}
// 或者注册时拒绝 NULL
if (obs == NULL) return false;
```

## 7. 最佳实践

1. **能静态就静态**。数组版满足 90% 需求，别上来就写链表。
2. **注册返回值检查**。`if (!register(x)) { /* 处理满的情况 */ }`——不要静默失败。
3. **注销要对称**。每个 `register` 都要有对应的 `unregister`——在观察者的 `deinit` 里调。
4. **通知函数要短**。`notify_all` 里只做 `for` 遍历 + 函数调用——不要在通知函数里写业务逻辑。
5. **回调不能阻塞**（非 RTOS 版）。如果观察者是函数回调（4.1-4.3），回调中不能有死循环、不能调 `vTaskDelay`、不能等信号量。
6. **ISR -> 队列 -> 任务**是嵌入式最安全的通知链路。如果你不确定回调的安全边界，走这条路。
7. **命名清晰区分同步/异步**：`batt_notify_sync` vs `batt_notify_from_isr`，调用者一看就知道约束。

## 8. 练习

**练习 1：GPIO 按键观察者（基础）**
一个 GPIO 按键，按下时需要通知 3 个模块：LED 控制器（切换灯状态）、蜂鸣器（短鸣一声）、事件日志（记录按键事件）。
要求：
- 用静态数组版实现
- 支持注册/注销
- ISR 中只投递事件到队列，任务中通知观察者

**练习 2：多传感器数据融合（进阶）**
三个传感器（温度、湿度、气压）各自有独立的采样 ISR。一个气象站显示屏需要显示综合数据，一个 BLE 模块需要定期上报。每次任意传感器更新时，重新计算综合数据并推送给观察者。
要求：
- 用侵入式链表版实现（每个传感器是独立的 Subject）
- 实现标记删除机制，避免遍历中注销的问题
- 按优先级排序：显示屏先更新，BLE 后上报

**练习 3：CAN 总线消息分发（综合）**
参考你现有的 [[can总线]] CAN RX 任务架构。CAN RX 任务收到报文后，根据 CAN ID 分发给不同的应用模块（车身控制、仪表盘、诊断模块）。每个模块可以动态注册/注销自己关心的 CAN ID。
要求：
- 用 Pub-Sub 模式（EventBus + topic 路由）
- 结合 RTOS 队列，CAN RX 任务作为 EventBus
- 思考：这个场景为什么更适合 Pub-Sub 而不是 Observer？

```c
/* battery_monitor.h */
#ifndef BATTERY_MONITOR_H
#define BATTERY_MONITOR_H
#include <stdint.h>
#include <stdbool.h>

#define MAX_OBSERVERS 4

typedef enum {
    BATT_NORMAL,     // > 3.6V
    BATT_LOW,        // 3.3~3.6V
    BATT_CRITICAL,   // < 3.3V
} batt_level_t;

typedef void (*batt_observer_t)(batt_level_t level, int voltage_mv);

void  batt_init(void);
bool  batt_register_observer(batt_observer_t obs);
void  batt_unregister_observer(batt_observer_t obs);
void  batt_sample_trigger(void);  /* 模拟 ADC 采样完成 */

#endif
```

```c
/* battery_monitor.c */
#include "battery_monitor.h"

static batt_observer_t observers[MAX_OBSERVERS];
static int observer_count = 0;

void batt_init(void) {
    observer_count = 0;
}

bool batt_register_observer(batt_observer_t obs) {
    if (!obs || observer_count >= MAX_OBSERVERS) return false;
    observers[observer_count++] = obs;
    return true;
}

void batt_unregister_observer(batt_observer_t obs) {
    for (int i = 0; i < observer_count; i++) {
        if (observers[i] == obs) {
            observers[i] = observers[--observer_count];
            return;
        }
    }
}

static batt_level_t calc_level(int mv) {
    if (mv > 3600) return BATT_NORMAL;
    if (mv > 3300) return BATT_LOW;
    return BATT_CRITICAL;
}

static void notify_all(batt_level_t level, int mv) {
    for (int i = 0; i < observer_count; i++) {
        if (observers[i]) observers[i](level, mv);
    }
}

/* 模拟：每 1 秒被定时器触发一次 */
void batt_sample_trigger(void) {
    static int simulated_mv = 3800;  /* 模拟电压缓慢下降 */
    simulated_mv -= 35;
    if (simulated_mv < 3100) simulated_mv = 3800;

    batt_level_t level = calc_level(simulated_mv);
    notify_all(level, simulated_mv);
}
```

```c
/* main.c — 使用示例 */
#include <stdio.h>
#include "battery_monitor.h"

/* ===== 观察者 1: 显示屏 ===== */
void display_update(batt_level_t level, int mv) {
    const char *icon;
    switch (level) {
        case BATT_NORMAL:   icon = "[FULL]"; break;
        case BATT_LOW:      icon = "[LOW] "; break;
        case BATT_CRITICAL: icon = "[CRIT]"; break;
    }
    printf("[Display] %s %dmV\n", icon, mv);
}

/* ===== 观察者 2: BLE 通知 ===== */
void ble_battery_notify(batt_level_t level, int mv) {
    int percent = (mv - 3100) * 100 / 1100;  /* 3.1V~4.2V -> 0~100% */
    if (percent > 100) percent = 100;
    if (percent < 0)   percent = 0;
    printf("[BLE]     Notify phone: %d%%\n", percent);
}

/* ===== 观察者 3: 低功耗报警 ===== */
void low_power_alert(batt_level_t level, int mv) {
    if (level == BATT_CRITICAL) {
        printf("[Alert]   CRITICAL! Entering low power mode!\n");
    }
}

int main(void) {
    batt_init();
    batt_register_observer(display_update);
    batt_register_observer(ble_battery_notify);
    batt_register_observer(low_power_alert);

    printf("=== Battery Monitor Demo ===\n");
    for (int i = 0; i < 25; i++) {
        batt_sample_trigger();
    }
    return 0;
}
```

**从这个示例可以看到**：
- 一个 Subject（电池采样）-> 三个 Observer（显示/BLE/报警），独立注册，独立工作
- 添加第四个观察者只需写一个新函数 + 一行 `register`
- Subject 代码不需要任何修改

## 附录 B：Observer vs Pub-Sub 快速参考卡片

| | 用 Observer 当... | 用 Pub-Sub 当... |
|---|---|---|
| **通信对象** | 同一模块内的几个函数/模块 | 跨子系统、跨任务的多个模块 |
| **注册方式** | Observer 主动调用 `register` | Subscriber 向 EventBus 订阅 topic |
| **通知路径** | Subject -> Observer（直接调） | Publisher -> EventBus -> Subscriber |
| **耦合** | Subject 持有 Observer 列表 | 双方只知道 EventBus 的 topic 名 |
| **C 实现** | 函数指针数组/链表 | 消息队列 + topic -> handler 路由表 |
| **时序** | 同步（立即执行） | 异步（经过队列） |
| **典型例子** | 电池采样 -> 显示/BLE/报警 | CAN ID 0x101 -> 车身控制/仪表盘 |
