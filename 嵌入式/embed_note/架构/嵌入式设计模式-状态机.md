# 状态机模式

## 1. 一句话

> 用有限的状态和明确定义的转移规则驱动系统行为——禁止出现未定义的状态转移。

## 你已经会了，只是还没命名

在你看这篇笔记之前，你已经在多个项目里写了状态机——只是用的是 `switch-case` + `enum`，没有叫它"状态机"：

| 你写的代码 | 隐含的状态机 |
|---|---|
| [[ota升级方案]] 中的 `IDLE → RECEIVING → FINISHED/ERROR` | OTA 升级状态机 |
| [[can总线]] 中 CAN 控制器的 Error Active / Error Passive / Bus Off | CAN 错误状态机 |
| [[消抖算法的选择]] 中按键的按下/释放/消抖判断 | 按键消抖状态机 |
| BLE 回调中的连接/断开/发现服务 | BLE 连接状态机 |

**状态机不是新东西——是把你在项目里已经在做的事情命名化、规范化。**

## 2. 核心概念

```
                        事件 (Event)
                           │
                           ▼
    ┌─────────┐   转移 (Transition)    ┌─────────┐
    │  状态 A  │ ──────────────────→  │  状态 B  │
    │  State   │    [守卫条件]         │  State   │
    └─────────┘    [转移动作]          └─────────┘
         │                                  │
         └──── 进入动作 / 退出动作 ──────────┘
```

四个要素：

| 概念 | 英文 | 在 C 中的表示 | 示例 |
|---|---|---|---|
| **状态** | State | `enum` 值 | `OTA_IDLE`, `CAN_ERR_ACTIVE` |
| **事件** | Event | `enum` 值、中断标志、消息 ID | `EV_DOWNLOAD_START`, `EV_BYTE_RECEIVED` |
| **转移** | Transition | `(当前状态, 事件) → 下一状态` 的映射 | `(IDLE, EV_START) → RECEIVING` |
| **动作** | Action | 函数指针 | `ota_begin()`, `crc_verify()` |

**状态机的本质**：把"if 嵌套"变成"状态 × 事件的查表"。

举个最简单的例子——不用状态机 vs 用状态机：

```c
// 不用状态机 —— 用 flags 堆砌，状态爆炸时无法维护
if (in_download && crc_ok && !aborted && space_available) {
    apply_firmware();
}

// 用状态机 —— 所有合法路径显式定义
// 状态 = OTA_DOWNLOADING, 事件 = EV_DOWNLOAD_DONE
// → 查表得：下一状态 = OTA_VERIFYING, 动作 = crc_check()
```

## 3. Mealy vs Moore

两种经典状态机模型的区别在于**输出（动作）绑定在哪里**：

| | Moore | Mealy |
|---|---|---|
| **动作绑定** | 绑定在**状态**上 | 绑定在**转移**上 |
| **输出取决于** | 只看当前状态 | 当前状态 + 触发事件 |
| **进入动作** | 进入状态时执行 | 无（动作在边上） |
| **状态数** | 偏多（需要不同状态区分输出） | 偏少（同一状态不同事件可不同输出） |

```c
// Moore: 动作在状态里
void enter_ota_receiving() {
    esp_ota_begin();     // 进入这个状态就执行
    led_set(BLINK_FAST);  // 不管怎么进来的，LED 都闪
}

// Mealy: 动作在转移上
// (IDLE, EV_DOWNLOAD_START) → RECEIVING，动作 = esp_ota_begin()
// (ERROR, EV_RETRY) → RECEIVING，动作 = esp_ota_begin()
```

**嵌入式里的实际选择**：不必纠结——混合用。
- 进入/退出动作属于 Moore 风格（硬件初始化/反初始化天然适合）
- 事件驱动的状态跳转属于 Mealy 风格（协议解析天然适合）
- 实际代码里几乎都是混合体，纯粹的形式分类学术意义大于工程意义

## 4. C 语言三种实现（递进）

用同一个 UART 协议解析器作为例子，展示三种写法的演化。协议格式：

```
STX(0x02) | CMD(1B) | LEN(1B) | DATA[LEN] | CRC(1B) | ETX(0x03)
```

完整可运行代码见 [[#^appendix-a|附录 A]]。

### 4.1 方法一：嵌套 switch-case（你会用的）

```c
typedef enum {
    WAIT_STX, WAIT_CMD, WAIT_LEN, WAIT_DATA, WAIT_CRC, WAIT_ETX
} state_t;

typedef enum {
    EV_BYTE, EV_TIMEOUT
} event_t;

state_t sm_run(state_t s, event_t evt, uint8_t byte)
{
    switch (s) {
    case WAIT_STX:
        if (evt == EV_BYTE && byte == 0x02) return WAIT_CMD;
        break;
    case WAIT_CMD:
        if (evt == EV_BYTE) { cmd = byte; return WAIT_LEN; }
        if (evt == EV_TIMEOUT) return WAIT_STX;
        break;
    case WAIT_LEN:
        if (evt == EV_BYTE) { len = byte; idx = 0; return WAIT_DATA; }
        if (evt == EV_TIMEOUT) return WAIT_STX;
        break;
    case WAIT_DATA:
        if (evt == EV_BYTE) {
            buf[idx++] = byte;
            if (idx >= len) return WAIT_CRC;
        }
        if (evt == EV_TIMEOUT) return WAIT_STX;
        break;
    case WAIT_CRC:
        if (evt == EV_BYTE) {
            if (byte == calc_crc()) return WAIT_ETX;
            return WAIT_STX; // CRC fail → abort
        }
        break;
    case WAIT_ETX:
        if (evt == EV_BYTE && byte == 0x03) {
            dispatch_cmd();  // 包完整，处理
            return WAIT_STX;
        }
        break;
    default: return WAIT_STX;
    }
    return s; // 未处理的事件，保持当前状态
}
```

**优点**：直观，任何 C 程序员都能读懂。  
**缺点**：6 个状态已经 50 行，加一个状态要改多处；转移规则隐式嵌入代码，一眼看不出全局。

### 4.2 方法二：表驱动（推荐，主力方法）

```c
// 转移表条目
typedef struct {
    state_t next;
    void (*action)(uint8_t byte);
} transition_t;

// 声明每个状态×事件的处理函数
static void act_store_cmd(uint8_t b)  { cmd = b; }
static void act_store_len(uint8_t b)  { len = b; idx = 0; }
static void act_store_data(uint8_t b) { buf[idx++] = b; }
static void act_check_crc(uint8_t b)  { crc_ok = (b == calc_crc()); }
static void act_finish(uint8_t b)     { (void)b; dispatch_cmd(); }
static void act_ignore(uint8_t b)     { (void)b; }

// 核心：转移表 = 状态机全部规则的集中定义
static const transition_t sm[NUM_STATES][NUM_EVENTS] = {
    //                   EV_BYTE                          EV_TIMEOUT
    [WAIT_STX]  = { {WAIT_CMD, act_store_cmd},           {WAIT_STX,  NULL} },
    [WAIT_CMD]  = { {WAIT_LEN, act_store_len},           {WAIT_STX,  NULL} },
    [WAIT_LEN]  = { {WAIT_DATA,act_store_data},          {WAIT_STX,  NULL} },
    [WAIT_DATA] = { {WAIT_CRC, NULL},                    {WAIT_STX,  NULL} },
    [WAIT_CRC]  = { {WAIT_ETX, act_check_crc},           {WAIT_STX,  NULL} },
    [WAIT_ETX]  = { {WAIT_STX, act_finish},              {WAIT_STX,  NULL} },
};

// 一行代码执行状态机
state_t sm_run(state_t s, event_t e, uint8_t byte)
{
    const transition_t *t = &sm[s][e];
    if (t->action) t->action(byte);
    return t->next;
}
```

**表驱动的核心价值**：

- **所有转移规则集中在一张表里**，一目了然
- **加一个状态/事件**：改 enum、加一行表条目，不改逻辑代码
- **编译期检查**：`_Static_assert` 确保表完整

```c
_Static_assert(
    sizeof(sm) / sizeof(sm[0]) == NUM_STATES,
    "转移表行数不等于状态数"
);
```

**何时用表驱动**：状态 > 3 个、事件 > 2 种、或项目生命周期中规则会变。这是你在绝大多数场景下的首选。

### 4.3 方法三：函数指针 Per-State（复杂场景）

当每个状态内部逻辑差异极大（比如 `WAIT_DATA` 要维护索引/缓冲区，`WAIT_STX` 就判一个字节），函数指针封装得更好：

```c
typedef struct sm_state sm_state_t;
struct sm_state {
    const char *name;
    void (*enter)(void *ctx);
    void (*exit)(void *ctx);
    sm_state_t *(*handle)(void *ctx, event_t evt, uint8_t byte);
};

static sm_state_t *state_wait_stx(void *ctx, event_t evt, uint8_t byte)
{
    if (evt == EV_BYTE && byte == 0x02)
        return &state_wait_cmd;
    return &state_wait_stx; // stay
}

static sm_state_t *state_wait_data(void *ctx, event_t evt, uint8_t byte)
{
    parser_ctx_t *c = (parser_ctx_t *)ctx;
    if (evt == EV_BYTE) {
        c->buf[c->idx++] = byte;
        if (c->idx >= c->len) return &state_wait_crc;
    }
    if (evt == EV_TIMEOUT) return &state_wait_stx;
    return &state_wait_data;
}

// 派发
sm_state_t *sm_dispatch(sm_state_t *s, void *ctx, event_t evt, uint8_t byte)
{
    sm_state_t *next = s->handle(ctx, evt, byte);
    if (next != s) {
        if (s->exit)  s->exit(ctx);
        if (next->enter) next->enter(ctx);
    }
    return next;
}
```

**权衡**：灵活性最高，但代码量最大。只在以下情况用：
- 不同状态需要的内部数据差异大
- 状态需要进入/退出动作（HAL 初始化和反初始化）
- 状态数量固定但每个状态行为复杂

**大多数嵌入式场景，方法二（表驱动）就是最优解。**

## 5. 进阶：进入/退出/守卫动作

真实的嵌入式状态机不止"状态→状态"，还有 entry/exit/guard/action 四个组件：

| 概念 | 执行时机 | 嵌入式用途 |
|---|---|---|
| **entry** | 进入状态时 | HAL 初始化（gpio_config, timer_start） |
| **exit** | 离开状态时 | 资源释放（timer_stop, gpio_deinit） |
| **guard** | 转移前判断 | 条件检查（flash 空间够吗、电压正常吗） |
| **action** | 转移过程中 | 日志记录、通知其他模块 |

以你熟悉的 OTA 升级为例，带 entry/exit/guard 的表驱动：

```c
typedef enum {
    OTA_IDLE, OTA_DOWNLOADING, OTA_VERIFYING,
    OTA_APPLYING, OTA_REBOOTING, OTA_ERROR
} ota_state_t;

typedef enum {
    EV_DL_START, EV_DL_DONE, EV_CRC_OK, EV_CRC_FAIL,
    EV_APPLY_OK, EV_APPLY_FAIL, EV_RETRY, EV_ABORT
} ota_event_t;

// guard 函数 —— 返回 false 阻止转移
bool guard_space_available(void *ctx) { /* 检查 flash 分区剩余空间 */ return true; }
bool guard_partition_valid(void *ctx) { /* 检查 OTA 分区表有效性 */ return true; }

// entry/exit 函数 —— 状态的资源管理
void entry_downloading(void *ctx) { esp_ota_begin(/*...*/); led_set(BLINK_FAST); }
void exit_downloading(void *ctx)  { esp_ota_end(/*...*/); }
void entry_applying(void *ctx)    { esp_ota_set_boot_partition(/*...*/); }

typedef struct {
    ota_state_t next;
    bool (*guard)(void *ctx);
    void (*entry)(void *ctx);
    void (*exit)(void *ctx);
    void (*action)(void *ctx);
} ota_transition_t;

// 完整转移表 —— 包括 guard/entry/exit/action
static const ota_transition_t ota_sm[OTA_NUM_STATES][OTA_NUM_EVENTS] = {
    [OTA_IDLE][EV_DL_START] = {
        .next  = OTA_DOWNLOADING,
        .guard = guard_space_available,
        .entry = entry_downloading,
    },
    [OTA_DOWNLOADING][EV_DL_DONE] = {
        .next = OTA_VERIFYING,
        .exit = exit_downloading,
    },
    [OTA_VERIFYING][EV_CRC_OK] = {
        .next  = OTA_APPLYING,
        .guard = guard_partition_valid,
        .entry = entry_applying,
    },
    [OTA_VERIFYING][EV_CRC_FAIL] = { .next = OTA_ERROR },
    [OTA_ERROR][EV_RETRY] = {
        .next  = OTA_DOWNLOADING,
        .guard = guard_space_available,
        .entry = entry_downloading,
    },
    [OTA_APPLYING][EV_APPLY_OK]   = { .next = OTA_REBOOTING },
    [OTA_APPLYING][EV_APPLY_FAIL] = { .next = OTA_ERROR },
    // 未定义的转移默认保持当前状态（需在 sm_run 中处理）
};

ota_state_t ota_sm_run(ota_state_t s, ota_event_t e, void *ctx)
{
    const ota_transition_t *t = &ota_sm[s][e];
    if (t->guard && !t->guard(ctx))
        return s;  // guard 失败，状态不变
    if (t->exit)   t->exit(ctx);
    if (t->action) t->action(ctx);
    if (t->entry)  t->entry(ctx);
    return t->next;
}
```

entry/exit 的核心价值：无论从哪条路径进入 `OTA_DOWNLOADING`，都执行相同的 `entry_downloading()`；无论因什么原因离开，都执行相同的 `exit_downloading()`。这就是"状态保证的资源管理"——对应的就是 [[嵌入式设计模式总览|RAII 思想在 C 中的体现]]。


## 6. 层次状态机 HSM

当状态机大到一定程度，很多状态共享相同行为。例如：任何状态下收到 `EV_EMERGENCY_STOP` 都应跳转到 `ERROR`。平铺状态机需要在**每一行**重复定义，层次状态机通过父子嵌套解决：

```
        ┌────────────────────────────┐
        │        正常运行 (父状态)     │
        │  ┌──────────────────────┐  │
        │  │  IDLE                │  │
        │  │  RECEIVING           │  │  任何子状态收到 EV_EMERGENCY
        │  │  VERIFYING           │  │  都跳到 ERROR——只定义一次
        │  └──────────────────────┘  │
        └────────────────────────────┘
                       │ EV_EMERGENCY
                       ▼
                   ┌─────────┐
                   │  ERROR   │
                   └─────────┘
```

核心逻辑：当前状态不处理的事件，交给父状态处理：

```c
typedef struct hsm_state {
    const char *name;
    struct hsm_state *parent;   // 父状态指针 —— 核心
    void (*enter)(void *ctx);
    void (*exit)(void *ctx);
    struct hsm_state *(*handle)(void *ctx, uint32_t event);
} hsm_state_t;

// 递归向上查找处理器
hsm_state_t *hsm_dispatch(hsm_state_t *current, uint32_t event, void *ctx)
{
    hsm_state_t *s = current;
    while (s) {
        hsm_state_t *next = s->handle(ctx, event);
        if (next != NULL && next != s) {
            // 简化版：找到目标就跳。完整版需要计算共同祖先(LCA)、
            // exit 到共同祖先、再 enter 到目标——见 QP/C 源码。
            return next;
        }
        if (next == s) return s;  // 显式处理了，保持当前
        s = s->parent;  // 当前状态不处理，问父状态
    }
    return current;  // 顶层也没处理，保持不变
}
```

**嵌入式中 HSM 的实际价值**：将"全局规则"（紧急停止、超时复位、低功耗休眠）下沉到父状态，子状态只关注自己的业务逻辑。

**何时用 HSM**：状态 > 10 个且有明显的行为共享层级。状态 < 10 个时，平铺表驱动足够。纯 C 手写 HSM 代码量大，推荐用现成框架（QP/C）。

## 7. RTOS 中的状态机：Active Object 模式

状态机 + RTOS = **Active Object（活动对象）**：每个状态机运行在自己的 RTOS 任务中，通过队列接收事件。

```
   ISR / 其他任务                 Active Object 任务
  ──────────────────            ───────────────────
  xQueueSend(event)  ──────→    xQueueReceive(&evt)
  xTaskNotify(event)  ──────→    sm_run(state, evt)
  xEventGroupSetBits() ──────→   state = next_state
```

这和你在 [[can总线]] 中 CAN RX 任务的架构本质相同：

```c
// 你的 CAN RX 任务结构 —— 就是 Active Object 模式
void can_rx_task(void *param) {
    can_ctx_t *ctx = (can_ctx_t *)param;
    can_state_t state = CAN_IDLE;
    can_event_t evt;

    while (1) {
        if (xQueueReceive(ctx->rx_queue, &evt, portMAX_DELAY)) {
            state = can_sm_run(state, evt.type, evt.data, ctx);
        }
    }
}
```

通用 Active Object 骨架（可直接在项目中复用）：

```c
typedef struct {
    QueueHandle_t queue;
    sm_state_t    current_state;
    void          *ctx;  // 指向你的业务数据结构
} active_object_t;

// 所有 Active Object 共用同一个任务函数
void ao_task(void *param)
{
    active_object_t *ao = (active_object_t *)param;
    event_t evt;

    while (1) {
        if (xQueueReceive(ao->queue, &evt, portMAX_DELAY) == pdTRUE) {
            ao->current_state = sm_run(ao->current_state, evt.type,
                                       evt.data, ao->ctx);
        }
    }
}

// 向 Active Object 发事件（任务上下文或 ISR 上下文均可）
void ao_post(active_object_t *ao, event_t *evt, TickType_t timeout)
{
    xQueueSend(ao->queue, evt, timeout);
}

void ao_post_from_isr(active_object_t *ao, event_t *evt,
                      BaseType_t *pxHigherPriorityTaskWoken)
{
    xQueueSendFromISR(ao->queue, evt, pxHigherPriorityTaskWoken);
}
```

这样你的整个系统就是多个独立的状态机，各自在任务里跑，通过事件通信。结合你已有的 [[TI RTOS和FreeRTOS]] 中的任务/队列知识，这些你都已经会了——只是现在用"状态机"的语言重新描述了一遍。


## 8. 常见陷阱

| 陷阱 | 问题 | 对策 |
|---|---|---|
| **未定义转移** | 某个(状态,事件)组合没有处理，行为随机 | 转移表必须填满；`_Static_assert` 检查行列数 |
| **转移函数里阻塞** | 在 action 里调用 `vTaskDelay` 或等信号量 | 动作函数必须非阻塞，耗时操作用独立任务 |
| **事件丢失** | ISR 发事件用 `xQueueSend` 而非 `FromISR` | ISR 里用 `xQueueSendFromISR` / `xTaskNotifyFromISR` |
| **状态的隐含假设** | "退出 A 状态时 timer 一定是停的"——但忘了 error 路径 | 用 exit 动作统一清理，不依赖调用方记得清理 |
| **状态爆炸** | 每个组合都变成独立状态（如 `IDLE_WITH_WIFI`, `IDLE_NO_WIFI`） | 正交状态用独立变量（如 `bool wifi_connected`），不合并进状态枚举 |
| **忘记 reset** | 从 ERROR 恢复时 `buf_idx` 没清零 | entry/exit 动作中显式初始化/清理上下文 |
| **转移表放 RAM** | `transition_t sm[STATE][EVENT]` 没有 `static const`，浪费 RAM | 转移表始终用 `static const`，放 `.rodata` |

## 9. 最佳实践

1. **先画图，再写代码**。ASCII 图就够——画出所有状态和箭头，标上事件名。
2. **状态和事件用 `enum`**，不要用 `#define` 或 magic number。编译器能检查 switch 完整性。
3. **转移表始终 `static const`**。放在 ROM 里，不占 RAM。见 [[嵌入式存储器体系速查笔记]]。
4. **给状态和事件起字符串名**，用于调试打印：

```c
static const char *state_names[] = {
    [OTA_IDLE] = "IDLE",
    [OTA_DOWNLOADING] = "DOWNLOADING",
    // ...
};
ESP_LOGI("SM", "%s --(%s)--> %s",
         state_names[old], event_names[evt], state_names[new]);
```

5. **`_Static_assert` 校验表完整性**：

```c
_Static_assert(
    sizeof(sm_table) / sizeof(sm_table[0]) == NUM_STATES,
    "转移表行数不等于状态数"
);
```

6. **不要在状态机动作里阻塞**。动作函数应该是瞬时完成的，耗时操作用独立任务 + 事件通知。
7. **先实现 Error 状态**，再写正常流程。嵌入式系统 80% 的代码在处理的不是"正常"而是"异常"。
8. **同一个模块用同一种写法**。不要模块 A 用 switch-case、模块 B 用表驱动、模块 C 用函数指针——选一种并统一。

## 10. 练习

### 练习 1：形式化你的 OTA 状态机

打开 [[ota升级方案]]，把里面的流程画成正规的状态图：
- 列出**所有状态**（不只是 IDLE/RECEIVING/FINISHED，还有 ERROR、APPLYING、REBOOTING）
- 列出**所有事件**（download_start, chunk_received, all_received, crc_ok, crc_fail, apply_ok, apply_fail, retry, abort, timeout）
- 画一张 N×M 的转移表，填满所有格子。标记哪些转移是你**当前代码里没处理**的
- 提问：CRC 失败时你是回到 IDLE 还是 ERROR？重试次数在哪记录？

### 练习 2：CAN Bus-Off 恢复状态机

参考 [[can总线]] 中的 CAN 错误状态，用**表驱动**实现：
- 状态：`CAN_ERR_ACTIVE`, `CAN_ERR_PASSIVE`, `CAN_BUS_OFF`
- 事件：`EV_TEC_ABOVE_96`, `EV_TEC_ABOVE_255`, `EV_TEC_BELOW_128`, `EV_RECOVERY_TIMEOUT`
- 动作：`log_warning()`, `enter_bus_off()`, `start_recovery_timer()`
- 移植到你的 TI C2000 或 ESP32 CAN 代码里

### 练习 3：按键长按检测状态机

这是你能在 ESP32 开发板上 15 分钟跑起来的练习：
- 状态：`RELEASED`, `DEBOUNCE_PRESS`, `PRESSED`, `LONG_PRESS`, `DEBOUNCE_RELEASE`
- 事件：`EV_GPIO_HIGH`, `EV_GPIO_LOW`, `EV_TIMER_50MS`（消抖），`EV_TIMER_1S`（长按判定）
- 用表驱动实现，打印每次状态转移
- 扩展：加双击检测（新增 2 个状态）

## 附录 B：Mealy vs Moore 快速参考

```
      Moore: 动作绑定在状态上              Mealy: 动作绑定在转移上

    ┌──────────┐                        ┌──────────┐
    │  STATE_A  │──entry: init()         │  STATE_A  │
    │  action() │                        │           │
    └──────────┘                        └─────┬─────┘
         │ event                               │ event / action()
         ▼                                     ▼
    ┌──────────┐                        ┌──────────┐
    │  STATE_B  │──entry: deinit()      │  STATE_B  │
    │  action() │                        │           │
    └──────────┘                        └──────────┘
```

|       | Moore                                          | Mealy                         |
| ----- | ---------------------------------------------- | ----------------------------- |
| 动作写在哪 | 状态的 enter/exit 函数里                             | 转移表的 `.action` 字段里            |
| C 实现  | `if (next != s) { s->exit(); next->enter(); }` | `if (t->action) t->action();` |
| 适用场景  | 硬件状态管理、电源模式切换                                  | 协议解析、事件响应                     |
| 嵌入式里  | **两者混用**：entry 初始化外设，action 处理业务               |                               |

**一句话选择**：硬件资源生命周期用 Moore（entry/exit），纯逻辑跳转用 Mealy（转移动作）。实际项目里永远是混合体。
