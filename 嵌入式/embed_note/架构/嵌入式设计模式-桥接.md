# 嵌入式设计模式-桥接模式

## 概述

桥接模式（Bridge Pattern）将**抽象部分**（功能/业务逻辑）与**实现部分**（硬件/具体操作）分离，使两者可以独立变化。在嵌入式 C 中，它的标准形式就是 **ops 结构体 + handle 模式**——你在 ESP-IDF 驱动中每天都在用。

## 核心概念

- **抽象（Abstraction）**：高层业务接口，定义"做什么"（如 `sensor_read_temperature()`）
- **实现（Implementor）**：底层硬件操作，定义"怎么做"（如 I2C 读写 / SPI 读写）
- **分离**：抽象持有实现指针，运行时解耦，两者各有独立的继承/变体树
- **嵌入式形态**：`xxx_handle_t + xxx_ops_t` 结构体对

## 细节

### 为什么桥接在嵌入式中如此普遍

嵌入式系统的核心矛盾：**业务逻辑要求通用**，**硬件接口存在差异**。

| 层次 | 变化原因 | 示例 |
|---|---|---|
| 抽象层 | 产品功能迭代 | 温度采样从"直接读取"变为"多次采样取均值" |
| 实现层 | 芯片选型变更 | 传感器从 I2C 版换成 SPI 版，或 MCU 的 I2C 外设换成了不同的 IP |

桥接模式让这两层可以**独立演进**——换传感器不改算法，改算法不换传感器。

### 标准 C 语言实现：ops + handle

```c
// ============ 实现层接口（Implementor） ============
typedef struct {
    esp_err_t (*init)(void *hw_cfg);
    esp_err_t (*read)(void *hw_cfg, uint8_t reg, uint8_t *data, size_t len);
    esp_err_t (*write)(void *hw_cfg, uint8_t reg, const uint8_t *data, size_t len);
} sensor_io_ops_t;

// ============ 抽象层（Abstraction） ============
typedef struct {
    sensor_io_ops_t *io;        // 桥接点：指向具体实现
    void            *io_cfg;    // 实现层私有配置（I2C addr / SPI cs_pin ...）
    int32_t          offset;    // 抽象层自有状态（校准偏移）
    int32_t          scale;     // 抽象层自有状态（缩放系数）
} temp_sensor_t;

// 抽象层 API —— 只管"读温度"，不管"怎么读"
int32_t temp_sensor_read(temp_sensor_t *sensor) {
    uint8_t raw[2];
    sensor->io->read(sensor->io_cfg, 0x00, raw, 2);  // 委托给实现层
    return (raw[0] << 8 | raw[1]) * sensor->scale + sensor->offset;
}
```

### 不同的硬件实现

```c
// ============ 实现 A：I2C 接口 ============
static esp_err_t i2c_read(void *cfg, uint8_t reg, uint8_t *data, size_t len) {
    i2c_cfg_t *i2c = (i2c_cfg_t *)cfg;
    return i2c_master_read(i2c->bus, i2c->addr, reg, data, len);

sensor_io_ops_t i2c_sensor_ops = {
    .init  = i2c_sensor_init,

    .write = i2c_write,

    spi_cfg_t *spi = (spi_cfg_t *)cfg;
    // SPI 读写时序：先发寄存器地址，再读数据
    ...
}

sensor_io_ops_t spi_sensor_ops = {
    .init  = spi_sensor_init,
    .read  = spi_read,
    .write = spi_write,

```

### 运行时绑定

```c
// I2C 版本
temp_sensor_t i2c_sensor = {
    .io     = &i2c_sensor_ops,
    .io_cfg = &i2c_dev_cfg,
    .offset = 0,
    .scale  = 100,
};

// SPI 版本 —— 同一套 temp_sensor_t 类型，同一套 temp_sensor_read() API
temp_sensor_t spi_sensor = {
    .io     = &spi_sensor_ops,
    .io_cfg = &spi_dev_cfg,
    .offset = 0,
    .scale  = 100,
};

int32_t temp = temp_sensor_read(&i2c_sensor);  // 调 I2C
int32_t temp = temp_sensor_read(&spi_sensor);  // 调 SPI —— 上层无需区分
```

### 桥接 vs 适配器

这两个模式经常被混淆，边界很清晰：

| 维度 | 桥接模式 | 适配器模式 |
|---|---|---|
| **设计时机** | 设计阶段刻意分离抽象与实现 | 已有两个不兼容接口后做匹配 |
| **目的** | 解耦，让两者独立变化 | 转换接口，让不匹配的配合工作 |
| **层次** | 抽象与实现是主从关系 | 两边是对等关系 |
| **嵌入式实例** | sensor_ops_t + sensor_handle_t | `I2C 设备驱动 → I2C 总线驱动` 的包装 |
| **是否改变接口** | 不，是同一接口的不同实现 | 是，从源接口转换为目标接口 |

**判断方法**：如果你在问"怎么让我的上层代码不关心底层是 I2C 还是 SPI"——这是桥接。如果你在问"这个 I2C 触摸芯片的寄存器和我的 I2C 总线 API 参数对不上"——这是适配器。

### 桥接 vs 策略（Strategy）

桥接和策略的结构图几乎一样（都是持有指向另一层级的指针），区别在意图：

- **桥接**：分离**抽象与实现**，两者是结构性的、相对稳定的关联
- **策略**：分离**算法与上下文**，是行为性的、运行时频繁切换的

> 同一组 ops 如果包含 init/read/write/ioctl 这类硬件操作 → 桥接
> 同一组 ops 如果包含 compress/filter/encode 这类算法变体 → 策略

由于 C 语言的 ops + handle 模型无法表达意图区别，实际代码中两者往往共用同一套机械结构。不必纠结分类，重要的是你理解了"哪部分在变、哪部分不变，以及如何把它们拆开"。

### 在 ESP-IDF 中的体现

ESP-IDF 大量使用桥接模式：

- **`i2c_master_bus_handle_t` + `i2c_master_bus_config_t`**：抽象层不关心 GPIO 配置是哪个管脚
- **`spi_device_handle_t` + `spi_bus_config_t`**：抽象层不关心 SPI 用的是 VSPI 还是 HSPI
- **`adc_oneshot_unit_handle_t` + `adc_oneshot_chan_cfg_t`**：ADC 采样逻辑与通道配置分离

这些例子的共同点：上层调用 API 时传入 handle，handle 内部持有 ops 指针和配置数据——这就是 C 语言中的桥接。
