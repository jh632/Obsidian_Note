## 一、系统概述

### 1.1 系统角色定义

| 角色 | 说明 |

|------|------|

| **设备端** | 智能手表/采集器：固件预置WebSocket地址（如 `ws://collector.local:8080` 或固定IP，该地址可以更改）。上电自动连接Wi-Fi，自动建立WebSocket连接，完成注册与心跳，接收命令并上传采集数据 |

| **服务端** | 维护设备连接、处理命令、接收数据推送，同时提供HTTP API给管理端 |

| **管理端** | 通过HTTP接口查看设备列表/状态，并可向指定设备下发控制命令 |

**通信约定**：所有通信基于 **JSON + 十进制**，不再使用二进制包头与校验。

## 二、设备发现机制

> ⚠️ **重要**: 本章节规范设备与服务端的相互发现机制，是硬件工程师编写固件的核心依据。

### 2.1 网络环境要求

    网络要求:

    ✅ 所有设备在同一局域网（192.168.x.x / 10.x.x.x）

    ✅ 设备可访问服务端端口 8080 (TCP WebSocket)

    ✅ 服务端可访问设备端口 5353 (UDP mDNS)

    ✅ 防火墙允许多播 (224.0.0.251:5353)

### 2.2 发现方案概述

本协议支持**两种设备发现机制**，可同时启用：

| 方案 | 机制 | 服务端角色 | 设备角色 | 推荐度 |

|------|------|-----------|---------|--------|

| **方案A: mDNS/DNS-SD** | 多播DNS服务发现 | 主动查询/监听设备 | 广播服务 | ⭐⭐⭐⭐⭐ |

| **方案B: 设备主动注册** | 设备上电自动连接 | 被动接收注册 | 主动连接 | ⭐⭐⭐⭐⭐ |

### 2.3 方案A: mDNS/DNS-SD 服务发现（推荐）

> 此方案适合**服务端主动发现设备**的场景

#### 2.3.1 服务类型定义

设备需在局域网内广播以下 mDNS 服务：

```

服务类型: _psy-device._tcp.local.

服务名:   psy-device-<device_id>.local.

端口:     8080 (WebSocket端口，设备监听)

```

#### 2.3.2 设备端 mDNS 广播规范

**设备上电后 5 秒内开始广播 mDNS 服务**

```c

// 设备固件伪代码

void device_mdns_broadcast_start() {

    // 1. 连接 Wi-Fi 成功后

    // 2. 等待 2-5 秒网络稳定

    // 3. 开始 mDNS 广播

    mdns_service_add(

        service_type: "_psy-device._tcp.local.",

        hostname: "psy-device-<device_id>",

        port: 8080,

        txt_record: {

            "device_id": "<序列号>",

            "model": "H10",

            "fw_version": "2.1.3"

        }

    );

}

```

#### 2.3.3 TXT 记录字段

| 字段 | 类型 | 必填 | 说明 |

|------|------|------|------|

| device_id | string | ✅ | 设备唯一序列号 |

| model | string | ✅ | 设备型号 |

| fw_version | string | ✅ | 固件版本 |

| hw_version | string | ❌ | 硬件版本 |

#### 2.3.4 服务端查询规范

```bash

# 服务端定期查询局域网内设备

# 查询间隔: 每 10 秒扫描一次

# 超时时间: 5 秒

# 查询所有 _psy-device._tcp.local 服务

# 使用 avahi-browse 或 mdns-scan 工具

# 收到的设备列表格式:

# Device: psy-device-SN123456789._psy-device._tcp.local

#         IP: 192.168.1.100:8080

#         TXT: device_id=SN123456789, model=H10, fw_version=2.1.3

```

### 2.5 方案对比与选择

| 维度         | 方案A: mDNS        | 方案B: 主动注册          |
| ---------- | ---------------- | ------------------ |
| **发现方向**   | 服务端 → 设备         | 设备 → 服务端           |
| **设备复杂度**  | 需要 mDNS 库        | 仅需 TCP / WebSocket |
| **服务端复杂度** | 需要 mDNS 查询       | 简单                 |
| **适用场景**   | 服务端需要管理设备列表、离线检测 | 设备数量固定，设备自主管理连接    |
| **推荐度**    | ⭐⭐⭐⭐⭐            | ⭐⭐⭐⭐⭐              |
| **建议**     | 服务端实现            | 设备端实现              |

两种方案均可采用，推荐根据实际网络环境和设备能力选择：若服务端需要主动发现动态设备，选 mDNS；若设备数量少且固定，主动注册更轻量。

 建议: 同时实现两种方案，服务端优先使用 mDNS 发现设备，

           设备端支持主动注册作为备份。

2.6 硬件工程师注意事项

    1. mDNS 实现

       ├─ 推荐库: lwIP (mDNS), Avahi (Linux), Bonjour (macOS/iOS)

       ├─ 设备上电后 5 秒内开始广播

       ├─ 服务类型: _psy-device._tcp.local

       └─ 保持广播直到设备关机

    2. 设备ID 生成

       ├─ 必须是全局唯一

       ├─ 建议: 芯片MAC地址 + 随机数哈希

       └─ 示例: SN123456789 (烧录在Flash)

    3. 网络超时

       ├─ mDNS 查询超时: 5 秒

       ├─ WebSocket 连接超时: 10 秒

       └─ 重试间隔: 指数退避 (1s → 2s → 4s → ... → 60s)

    4. 固件预置信息

       ├─ Wi-Fi SSID / 密码

       ├─ 服务端地址 (collector.local 或固定IP)

       ├─ 设备序列号 (device_id)

       └─ mDNS 服务类型

## 六、数据通道规格（硬指标）

> ⚠️ **重要**: 以下为硬件真实规格，是协议设计的基础依据。

### 4.1 支持的数据通道总览

| 通道 | 传感器 | 采样率 | 说明 |

|------|--------|--------|------|

| PPG | 光电容积 | 25 Hz | 心率、血氧 |

| EDA | 皮电 | 50 Hz | 皮肤电活动/SCR |

| IMU | 惯性单元 | 25 Hz | 加速度、陀螺仪 |

| ENV | 环境 | 1 Hz | 温度、湿度 |

### 4.2 数据打包策略

为减少网络开销，采用 **200ms 固定打包间隔**：

|通道|采样率|每包样本数|样本间隔|包频率|每秒包数|
|---|---|---|---|---|---|
|PPG|25 Hz|5|40 ms|200 ms|5|
|EDA|50 Hz|10|20 ms|200 ms|5|
|IMU|25 Hz|5|40 ms|200 ms|5|
|ENV|1 Hz|1|1000 ms|1000 ms|1|

⚠️ 注意: 所有200ms数据包在时间上对齐，但ENV独立1s周期

### 4.3 数据包总带宽估算

|通道|包大小（估算）|包/秒|带宽占用|
|---|---|---|---|
|PPG|~200 B|5|~1.0 KB/s|
|EDA|~300 B|5|~1.5 KB/s|
|IMU|~400 B|5|~2.0 KB/s|
|ENV|~100 B|1|~0.1 KB/s|
|**总计**|||**~4.6 KB/s**|

    ✅ 全开(4通道): 约 5 KB/s ≈ 40 Kbps

    ✅ 典型场景(PPG+EDA+IMU): 约 4.5 KB/s ≈ 36 Kbps

## 七、WebSocket 接口详细定义

### 5.1 通用消息格式

所有消息均为 JSON 对象。**每条消息必须包含 device_id**。

•      消息总览

| 字段        | 类型     | 必填  | 说明              | 示例值 / 备注                             |
| --------- | ------ | --- | --------------- | ------------------------------------ |
| device_id | string | ✓   | 设备唯一序列号         | "SN123456789"                        |
| req_id    | int    | ✓   | 请求序列号，用于匹配响应    | 1                                    |
| cmd       | string | ✓   | 命令名称            | "register"                           |
| params    | object | 可选  | 命令参数，结构随 cmd 变化 | {"model":"H10","fw_version":"2.1.3"} |
|           |        |     |                 |                                      |

**示例 JSON：**

json

{

  "device_id": "SN123456789",

  "req_id": 1,

  "cmd": "register",

  "params": { "model": "H10", "fw_version": "2.1.3" }

}

•      响应消息：

|字段|类型|必填|说明|示例值 / 备注|
|---|---|---|---|---|
|device_id|string|✓|设备唯一序列号|"SN123456789"|
|req_id|int|✓|对应请求的序列号|1|
|cmd|string|✓|响应命令名，通常为 <原命令>_resp|"register_resp"|
|code|int|✓|状态码，0 表示成功|0|
|data|object|可选|返回的数据，结构随命令变化|{"server_time": 1715001234567}|

**示例 JSON：**

json

{

  "device_id": "SN123456789",

  "req_id": 1,

  "cmd": "register_resp",

  "code": 0,

  "data": { "server_time": 1715001234567 }

•      命令推送消息：

|字段|类型|必填|说明|示例值 / 备注|
|---|---|---|---|---|
|device_id|string|✓|设备唯一序列号|"SN123456789"|
|req_id|int|✓|请求序列号，设备响应需带回此值|999|
|type|string|✓|固定为 "command"，表示命令推送|"command"|
|cmd|string|✓|命令名称，如 start_collect|"start_collect"|
|params|object|可选|命令参数，按命令定义|{"duration": 60, "channel": "ppg"}|

**示例 JSON：**

json

{

  "device_id": "SN123456789",

  "req_id": 999,

  "type": "command",

  "cmd": "start_collect",

  "params": { "duration": 60, "channel": "ppg" }

}

设备收到命令推送后，应使用相同的 req_id 发送响应消息（响应格式参见上一节），以确认命令执行结果。

•      数据推送消息：

|字段|类型|必填|说明|示例值 / 备注|
|---|---|---|---|---|
|device_id|string|✓|设备唯一序列号|"SN123456789"|
|type|string|✓|数据类型，如 ppg / eda / imu / env|"ppg"|
|timestamp|int64|✓|毫秒级时间戳，表示采样点起始时间|1715001234567|
|samples|array|✓|采样数据数组，结构与通道相关|[1452, 1480, 1501, ...]|
|req_id|-|无|数据推送无需 req_id，服务端不应应答|省略该字段|

**示例 JSON：**

json

{

  "device_id": "SN123456789",

  "type": "ppg",

  "timestamp": 1715001234567,

  "samples": [1452, 1480, 1501, 1498, 1512]

}

数据推送为单向传输，服务端不返回响应，设备不应期待 req_id 或应答。

### 5.2 注册（register）

**设备 WebSocket 连接建立后必须第一个发送。**

json

// 请求

{

  "device_id": "SN123456789",

  "req_id": 1,

  "cmd": "register",

  "params": {

    "model": "H10",

    "hw_version": "1.0",

    "fw_version": "2.1.3",

    "battery_level": 85

  }

}

// 响应

{

  "device_id": "SN123456789",

  "req_id": 1,

  "cmd": "register_resp",

  "code": 0,

  "data": {

    "server_time": 1715001234567,

    "session_id": "uuid-xxxx"

  }

}

| code | 含义 |

|------|------|

| 0 | 成功 |

| 1 | 设备已存在（旧连接被踢） |

| 2 | 黑名单 |

**设备收到 code=0 后进入心跳与命令监听状态。**

### 5.3 心跳（ping）

```json

// 请求 (设备每30秒发送)

{

  "device_id": "SN123456789",

  "req_id": 2,

  "cmd": "ping",

  "params": {}

}

// 响应

{

  "device_id": "SN123456789",

  "req_id": 2,

  "cmd": "ping_resp",

  "code": 0,

  "data": { "server_time": 1715001235000 }

}

```

**规则**：

- 设备应每隔 **30秒** 发送一次 ping

- 服务端回复 pong

- 服务端若 **90秒** 未收到任何消息则应报警，应用选择断开或者重连

### 5.5 设备主动推送的数据包

**数据流程说明**：

1. 设备端每 200ms 推送一次原始采样数据（数组格式）

2. 服务端接收后进行后处理（计算衍生指标、统计值）

3. 服务端保存到 jsonl 文件，并推送给前端

**统一规则**：

- 所有数据包采用 **200ms 固定打包间隔**（ENV 除外）

- `timestamp` 为包首样本的 Unix 时间戳（秒，浮点数）

- 数据推送不需要 `req_id`，服务端无需应答

数据包顶层格式

{

  "device_id": "SN123456789",

  "type": "ppg | eda | imu | env",

  "timestamp": 1778503238.9285774   // Unix 时间戳（秒，浮点数）

}

```

**服务端存储/转发格式**（服务端推送给前端时，包含衍生指标，此处和下位机无关）：

```json

{

  "timestamp": 1778503238.9285774,

  "device_id": "SN123456789",

  "channel": "ppg",

  "test_id": "111",

  "user_id": "222",

  "test_num": 333,

  // ... 通道具体数据（见下方各节）

}

```

#### 5.5.1 PPG 光电容积脉搏波（type="ppg"）

| 参数    | 值                   |     |
| ----- | ------------------- | --- |
| 采样率   | 25 Hz               |     |
| 每包样本数 | 25 个（1秒窗口）          |     |
| 采样间隔  | 40 ms               |     |
| 推送间隔  | 200 ms（5 包/秒，5 点/包） |     |

**设备推送格式（原始数据）**：

```json

{

  "device_id": "SN123456789",

  "type": "ppg",

  "timestamp": 1778503238.9285774,

## 十二、服务端实现要点

### 10.1 设备映射管理

```text

维护内存映射：

device_id -> {

  conn: WebSocket连接,

  last_heartbeat: timestamp,

  registered: bool,

  collecting: bool,

  data_stream_mask: int,

  ...

}

```

### 10.2 处理相同 device_id 重复连接

- 如果新连接携带的 `device_id` 已存在活跃连接，服务端关闭旧连接（发送关闭帧）并接受新连接

- 注册响应中 `code=0` 表示旧连接已被踢

### 10.3 心跳超时

```

┌─────────────────────────────────────────────────────────────────────────────┐

│                           心跳超时检测流程                                   │

└─────────────────────────────────────────────────────────────────────────────┘

    设备发送消息 ──▶ 更新 last_heartbeat ──▶ 重置超时计数器

                                            │

                                            ▼

                              ┌─────────────────────────┐

                              │   定时器每10秒扫描一次    │

                              └─────────────────────────┘

                                            │

                                            ▼

                              ┌─────────────────────────┐

                              │ 检查: now - last_beat   │

                              │      > 90s ?            │

                              └─────────────────────────┘

                                            │

                          ┌─────────────────┴─────────────────┐

                          │                                   │

                         Yes                                  No

                          │                                   │

                          ▼                                   ▼

              ┌───────────────────┐                  ┌───────────────────┐

              │ 关闭WebSocket    │                  │   继续监听        │

              │ 清理设备映射      │                  │   无需操作        │

              └───────────────────┘                  └───────────────────┘

```

- 每收到任何消息（ping、业务请求、数据推送）都更新 `last_heartbeat`

- 定时器每 **10秒** 扫描一次

- 若 `now - last_heartbeat > 90s`，主动关闭 WebSocket 并清理映射

- 设备端应实现自动重连，重连后重新注册

      │                          │                            │    ┌────────────┐

      │                          │─── 订阅事件 ──────────────▶│──▶│ 管理端WebSocket

      │                          │                            │    └────────────┘

```

- 服务端可将收到的 PPG/EDA/IMU/ENV 数据包存入时序数据库（如 InfluxDB）

- 或转发给消息队列（如 Kafka）

- 对管理端 WebSocket 订阅者，可根据订阅条件转发数据包（仅 `data_packet` 事件）

### 10.5 命令下发可靠性

- 服务端对每个下发的命令生成唯一的 `req_id`（可跨设备递增）

- 若设备 **30秒** 内未回复，超时返回错误

- 设备可能已断开，应依赖重连

### 10.6 设备注册合法性校验

- 可维护白名单（预知 SN 列表）或动态允许（只要格式符合）

- 生产环境建议加上简单的 token 或密码（在 `register` 参数中加入 `"token": "xxx"`）

```json

// 建议的带Token注册请求

{

  "device_id": "SN123456789",

  "req_id": 1,

  "cmd": "register",

  "params": {

    "model": "H10",

    "hw_version": "1.0",

    "fw_version": "2.1.3",

    "battery_level": 85,

    "token": "预置的设备认证Token"

  }

}

## 状态码(code)汇总表

| 场景 | code值 | 含义 |

|------|--------|------|

| **通用** | 0 | 成功 |

| **通用** | 1 | 已存在/未注册 |

| **通用** | 2 | 黑名单 |

| **通用** | 255 | 错误/失败 |

| **register** | 0 | 成功 |

| **register** | 1 | 设备已存在(旧连接被踢) |

| **register** | 2 | 黑名单 |

| **start_collect** | 0 | 成功，开始采集 |

| **start_collect** | 1 | 设备未注册 |

| **start_collect** | 2 | 内存不足 |

| **start_collect** | 3 | 掩码无效 |

| **start_collect** | 4 | 设备正在采集中 |

| **start_collect** | 255 | 初始化失败 |

| **stop_collect** | 0 | 成功，停止采集 |

| **stop_collect** | 1 | 设备未在采集状态 |

| **stop_collect** | 255 | 失败 |

| **shutdown** | 0 | 成功，即将关机 |

| **shutdown** | 1 | 正在采集中，需先停止 |

| **shutdown** | 255 | 错误 |
