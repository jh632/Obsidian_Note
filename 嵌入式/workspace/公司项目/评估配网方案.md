# PSA 穿戴设备 WiFi 配网方案评估

**长按按键 AP 配网 vs 屏幕 UI 配网**  
评估日期：2026-06-10  
硬件：ESP32-S3 + CO5300 LCD (410×502) + GPIO45 按键 ×1

## 一、方案对比总览

以下表格从代码量、用户体验、资源消耗、维护复杂度等方面对比两种方案：

| 维度 | AP 配网（推荐） | UI 配网 |
|------|----------------|---------|
| 代码量 | ~100 行 C | 300-500 行 C + SquareLine Studio 改 UI |
| 新增文件 | `provisioning.c` + `provisioning.h` | 新增 2+ 个 UI 屏幕 + 输入逻辑 + 键盘组件 |
| 输入方式 | 手机浏览器键盘 | 单按键 "拼音/九宫格" 逐字符选择 |
| 输入完成时间 | 30 秒 | 3-5 分钟（SSID + 密码） |
| 用户门槛 | 连 WiFi → 浏览器 | 学习按键输入法 |
| 内存增量(峰值) | +6 KB | +15~20 KB（额外 LVGL 屏幕 + 键盘缓存） |
| 二手设备支持 | 任何有浏览器的手机 | 只能设备本机操作 |
| 配网范围 | 手机即可，不限距离 | 必须在设备前 |
| 调试难度 | 低（日志 + curl 即可测） | 中（需模拟按键序列） |
| 影响现有 UI | 否，完全独立模块 | 是，新增屏幕 + 按键状态机冲突 |

## 二、AP 配网（推荐方案）

### 2.1 核心思路

用户长按电源键 1 秒 → 关闭 STA 模式释放内存 → 启动 AP + HTTP 配网服务器 → 用户通过手机浏览器配置 WiFi → 配网成功后自动切换回 STA 模式。  
**关键约束**：空闲堆约 29KB，WiFi STA 已占用约 20KB。  
**核心原则**：不叠加 AP+STA，而是顺序复用内存。

### 2.2 内存策略

长按触发 → `wifi_ws_deinit()` 释放 STA（+~20KB）→ 开 AP 复用此内存 → HTTP server 额外 +~6KB → 峰值堆约 23KB，安全。  
配网成功 → 关 AP + HTTP → `wifi_ws_init()` 以新凭据启动 STA。

### 2.3 详细设计

| 维度 | 说明 |
|------|------|
| 做什么 | 新增配网模块，复用已有的 `wifi_ws_deinit()` / `wifi_ws_init()` 在 STA 和 AP 模式间切换。长按按键触发，完成后自动切回。 |
| 对外接口 | `provisioning_start()` — 关STA→开AP→起HTTP Server<br>`provisioning_stop()` — 关AP→开STA（配网成功时自调用）<br>`provisioning_is_active()` — 查询当前是否在配网模式 |
| 依赖硬件和配置 | GPIO45 按键（现有）、ESP32-S3 WiFi（内置）、NVS（已有 `wifi_ws_config_save/load`）、`esp_http_server`（ESP-IDF内置）、`esp_netif` + `esp_wifi` |
| 常见问题排查 | 长按无反应→回调是否注册、GPIO45电平<br>AP搜不到→`esp_wifi_start()`返回值、信道干扰<br>配网页打不开→`httpd_start()`、DHCP 192.168.4.x<br>提交后不重启→`wifi_ws_config_save()`返回值 |
| 测试方法 | 手动：手机连 PSA-Config → 浏览器配网<br>调试：`curl -X POST http://192.168.4.1/config ...` |
| 维护要点 | 纯新增模块，不改现有文件（除 `driver_key.c` 加一行回调）；HTML在flash中，改页面无需改C逻辑 |

## 三、UI 配网（备选方案）

### 3.1 核心困难

显示屏不算大（410×502），LVGL 已就绪。但用户屏幕过小，键盘输入困难，定制键盘设计较为困难。

### 3.2 详细设计

| 维度 | 说明 |
|------|------|
| 做什么 | 利用 LVGL + LCD 在设备本机实现 WiFi 配网界面。核心困难：只有一个按键作为输入设备（GPIO45）。 |
| 对外接口 | `app_ui ops` 增加 `set_wifi_config_page` / `set_keyboard_page` 接口<br>`provisioning.h` 同 AP 方案（底层WiFi切换一致） |
| 依赖硬件和配置 | CO5300 LCD (410×502) ✅<br>LVGL v8.4 ✅<br>GPIO45 按键 ×1 ⚠️ 致命瓶颈<br>无触摸屏 ❌ 无键盘 ❌ |
| 常见问题排查 | 输入到一半想取消→需要额外逻辑<br>SSID特殊字符→键盘需支持 `!@#$%^&`<br>光标与输入不一致→LVGL焦点+按键状态机同步<br>配网页与主UI切换卡顿→内存压力 |
| 测试方法 | 手动：反复按键验证每个字符能否正确选中<br>自动化：需模拟按键序列脚本，难以覆盖<br>验证边界：空SSID、64位密码、特殊字符 |
| 维护要点 | UI屏幕文件（SquareLine Studio生成）+ C逻辑两处维护<br>按键状态机和主UI自动轮播可能冲突<br>输入法逻辑（移动/选中/退格/提交）复杂易出边界问题 |

## 四、结论与建议

| 维度 | AP 配网 | UI 配网 |
|------|---------|---------|
| 推荐 | ✔ 推荐 | ✘ 不推荐 |
| 核心理由 | 手机浏览器天然有键盘，用户 30 秒完成配网 | 单按键输入 WiFi 密码体验灾难，3-5 分钟 |
| 代码量 | 100 行 | 300-500 行 + 屏幕文件 |
| 开发周期 | 6-8 小时 | 2-3 天 |
| 测试难度 | 低（curl 即可） | 高（需模拟几十次按键序列） |
