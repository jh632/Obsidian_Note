## 1 时间戳格式
文档提到 timestamp 是“毫秒级 int64” ,后面5.5又提到`timestamp` 为包首样本的 Unix 时间戳（秒，浮点数）

# 2 连接方式确定
两套相反的拓扑：
- 一套是“设备连服务端”采用websocket连接
- 一套是“服务端连设备”采用mdns广播(espidf内置mdns库)  
    这两个不能同时作为主协议依据，必须先定唯一权威方案。
# 3 数据包 JSON 结构不统一
5.1通用数据格式中数据推送格式是

| 字段        | 类型     | 必填  | 说明                           | 示例值 / 备注                |
| --------- | ------ | --- | ---------------------------- | ----------------------- |
| device_id | string | ✓   | 设备唯一序列号                      | "SN123456789"           |
| type      | string | ✓   | 数据类型，如 ppg / eda / imu / env | "ppg"                   |
| timestamp | int64  | ✓   | 毫秒级时间戳，表示采样点起始时间             | 1715001234567           |
| samples   | array  | ✓   | 采样数据数组，结构与通道相关               | [1452, 1480, 1501, ...] |
| req_id    | -      | 无   | 数据推送无需 req_id，服务端不应应答        | 省略该字段                   |
# 4 采样包大小定义冲突
4.2打包策略写的

| 通道  | 采样率   | 每包样本数 | 样本间隔    | 包频率     | 每秒包数 |
| --- | ----- | ----- | ------- | ------- | ---- |
| PPG | 25 Hz | 5     | 40 ms   | 200 ms  | 5    |
| EDA | 50 Hz | 10    | 20 ms   | 200 ms  | 5    |
| IMU | 25 Hz | 5     | 40 ms   | 200 ms  | 5    |
| ENV | 1 Hz  | 1     | 1000 ms | 1000 ms | 1    |
**但是5.5 ppg**

| 参数    | 值                   |
| ----- | ------------------- |
| 采样率   | 25 Hz               |
| 每包样本数 | 25 个（1秒窗口）          |
| 采样间隔  | 40 ms               |
| 推送间隔  | 200 ms（5 包/秒，5 点/包） |

 # 5 状态/响应语义有歧义
register 的 code=1 定义成“设备已存在（旧连接被踢）”，见 474 (line 474)。问题是：  
如果旧连接被踢，新连接到底算成功还是失败？  
从设备侧实现看，这个状态不好处理，因为它既像成功，又像错误。
